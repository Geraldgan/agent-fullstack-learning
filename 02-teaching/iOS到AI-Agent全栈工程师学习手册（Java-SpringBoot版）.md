# iOS 工程师 → AI Agent 全栈工程师学习手册
## Java / Spring Boot 主线版

> 适合对象：有 iOS 开发经验，准备补齐 Java 后端、AI Agent、前端与工程化能力，目标是做出真正可交付 AI 产品的人。

---

# 1. 你到底要转成什么样的人

你的目标，不应该只是“会调用大模型 API”，也不应该只是“从 iOS 转成 Java 后端”。

更准确地说，你要转成的是：

> **能够独立设计、开发、部署、优化 AI Agent 系统的全栈工程师。**

一个完整的 AI Agent 系统通常包含：

- **客户端入口**：App、Web、聊天界面、飞书/企业微信/IM
- **后端服务**：Java / Spring Boot，承接业务逻辑、权限、数据、任务编排
- **AI 能力层**：LLM、Prompt、RAG、Tool Calling、Workflow、Agent
- **数据层**：MySQL / PostgreSQL、Redis、向量库、对象存储
- **工程层**：日志、监控、限流、评测、成本控制、部署

所以你的升级路径不是：

- iOS → 学点 AI

而是：

- iOS 工程师 → 会后端 → 会 AI Agent → 会交付完整系统

## 阶段 B：学会把 LLM 接入 Java 系统

### 目标

从传统后端升级成 AI 后端。

### 必学内容

#### LLM 的基本工作方式

你不用一开始研究 Transformer 数学细节，但必须理解：

- 模型处理的是 token，不是“真正理解世界”
- 它本质上是在根据上下文预测后续 token
- 它不是数据库，也不是搜索引擎
- 它有概率性，有幻觉
- 它对上下文极其敏感

#### API 调用

你必须会：

- 调 chat / responses 类接口
- 传 system / user / assistant 消息
- 控制 temperature、max tokens
- 处理流式输出
- 处理超时、重试、限流

#### Prompt 设计

要掌握：

- system prompt 的角色
- task prompt 的结构化表达
- 如何明确输入 / 输出格式
- few-shot 示例
- 约束条件设计
- 如何减少模型跑偏

#### 结构化输出

工程里非常关键。

你要会让模型输出：

- JSON
- 指定字段
- 固定 schema
- 稳定可解析结果

#### 上下文管理

你要理解：

- 为什么不能无限堆聊天记录
- 如何裁剪历史消息
- 如何做摘要
- 如何保留重要状态
- 如何构造会话级 memory

## 阶段 C：掌握 RAG 和知识库

### 目标

让系统不只靠模型“记忆”，而是能够查资料再回答。

### 什么是 RAG

RAG = Retrieval-Augmented Generation。

本质上是：

> 先检索相关资料，再把资料作为上下文交给模型生成答案。

它适合：

- 企业知识库
- FAQ
- 文档问答
- SOP 流程
- 基于资料的问答场景

### RAG 的核心流程

1. 文档导入
2. 文档切块
3. embedding 向量化
4. 存入向量库
5. 用户提问
6. 检索相关片段
7. 拼接上下文
8. 模型生成答案

### 必学内容

#### 文档切块

要理解：

- 为什么要切块
- chunk size
- overlap
- 按段落切
- 按标题切
- 按语义切

#### Embedding

要知道：

- embedding 是语义向量表示
- query 和 document 都需要向量化
- 常见相似度如 cosine similarity

#### 向量库

要理解它的核心职责：

- 存向量
- 召回相似内容
- 支持 metadata filter

#### 检索优化

RAG 不只是“接个向量库”：

- topK
- metadata filter
- rerank 重排
- hybrid search 混合检索
- query rewrite 查询改写

#### 引用与可追溯

好的知识库系统应该：

- 给出引用来源
- 能回溯原文片段
- 尽量减少胡编乱造

## 阶段 D：从“AI 接口调用”升级到 Agent

### 目标

理解 Agent 到底是什么，不停留在“聊天机器人”阶段。

### Workflow 和 Agent 的区别

#### Workflow

Workflow 是预先设计好的流程。

例如：

1. 用户上传文档
2. 系统提取文本
3. 模型摘要
4. 结构化输出
5. 返回结果

特点：

- 步骤固定
- 稳定可控
- 更适合企业流程

#### Agent

Agent 是目标驱动下的动态执行。

例如：

“帮我分析一个客户并起草一封跟进邮件。”

系统可能会：

1. 查客户资料
2. 查历史聊天
3. 搜公司背景
4. 总结关键信息
5. 起草邮件
6. 等人工确认

特点：

- 步骤不完全固定
- 可动态选工具
- 灵活但更难控

### 工程上的推荐姿势

多数真实项目不是纯 Agent，而是：

> **Workflow + 局部 Agent 能力**

即：

- 大流程用代码控制
- 某些复杂判断交给模型

这是最稳的方案。

# 5. Tool Calling / Function Calling 是什么

这是 Agent 工程最关键的能力之一。

它的本质不是“模型真的会执行工具”，而是：

> 模型根据你提供的工具描述，决定是否要调用某个函数，并输出调用参数；真正执行工具的是你的程序。

例如你定义：

- `searchDocs(query)`
- `getWeather(city)`
- `createTask(title, dueDate)`

模型可能会输出：

- 调用 `getWeather(city="上海")`

然后后端做的事是：

1. 解析工具调用
2. 执行 Java 方法或外部 HTTP 请求
3. 得到结果
4. 把结果放回上下文
5. 再让模型生成最终回答

### 为什么重要

没有 Tool Calling，模型只能“说”；
有了 Tool Calling，模型才能“做”。

### Java / Spring Boot 如何落地

你通常会做：

- 定义工具 schema
- 编写工具执行器
- 设计统一 tool response 格式
- 建立 Agent loop
- 做超时、重试、权限控制

# 6. Memory 到底是什么

Memory 没那么神秘，工程里通常分为三类：

## 6.1 短期记忆

当前会话里的上下文，比如：

- 最近几轮对话
- 当前任务状态
- 当前会话内的重要事实

## 6.2 长期记忆

跨会话保留的信息，比如：

- 用户偏好
- 常用项目
- 历史任务
- 稳定背景资料

## 6.3 工作记忆

任务执行中的中间状态，比如：

- 已经调用过哪些工具
- 已获取哪些信息
- 哪些子任务已完成

### 关键理解

Memory 不是“把所有聊天记录永久保存”。
真正难的是：

- 存什么
- 什么时候更新
- 怎么检索
- 什么时候清理

否则只会变成脏数据堆积。

# 7. Planning 是什么

Planning 就是任务拆解。

例如用户说：

“帮我做一个竞品调研并给出总结。”

系统可能拆成：

1. 列出竞品
2. 获取资料
3. 分维度比较
4. 总结优缺点
5. 输出文档

### 规划能力的价值

复杂任务通常不是一次 prompt 能解决，而是：

- 先拆任务
- 逐步执行
- 检查结果
- 必要时修正路径

### 但别神化它

很多工程场景下，不需要“超强智能规划”，而是：

- 规则化拆解
- 模板化步骤
- 少量 LLM 决策

这已经够用了。

# 8. Java / Spring Boot 在 AI Agent 里的位置

很多教程偏 Python，但你走 Java 路线完全可行。

## 8.1 Java 的优势

- 企业级生态成熟
- Spring Boot 工程化能力强
- 权限、事务、数据库、接口治理都很完整
- 更适合接企业内部系统
- 团队协作与生产环境部署更稳

## 8.2 Java 的挑战

- AI 新框架往往先在 Python 出现
- 一些新功能示例偏少
- 某些 AI SDK 的生态不如 Python 快

## 8.3 正确姿势

不要执着于“一切都必须 Java 写”。

更现实的架构一般有两种：

### 方案 A：纯 Java 主服务

- Spring Boot 负责主后端
- 直接调模型 API
- 集成知识库与工具系统

适合大多数应用型项目。

### 方案 B：Java 主服务 + Python AI 子服务

- Java 负责业务主线
- Python 负责某些 AI 特殊能力
- 二者通过 HTTP / MQ 协作

适合后期复杂场景。

### 当前建议

你现阶段优先走：

> **方案 A：Spring Boot 作为主服务，先把完整链路跑通。**

# 10. 你最容易踩的坑

## 坑 1：以为会调模型 API 就算会 AI 工程

不是，那只是起点。

## 坑 2：沉迷 Prompt，不补后端和系统能力

结果只能做 demo，不能上线。

## 坑 3：过早追逐复杂 Agent

一上来就 multi-agent / autonomous agent，最后难以控制、难以落地。

## 坑 4：忽略检索与数据质量

RAG 做不好，回答就会非常不稳定。

## 坑 5：只关心生成效果，不关心系统稳定性

真实项目里，稳定、可追踪、可恢复，通常比“偶尔惊艳”更重要。

## 坑 6：什么都学，主线不清

今天 LangChain，明天 AutoGen，后天再追一个新框架，最后脑子里没有系统图。

# 15. 学习方式建议

最有效的方法不是“看很多课”，而是：

> **一边做项目，一边补知识。**

推荐节奏：

1. 先定一个小项目
2. 做到卡住
3. 针对卡点补知识
4. 回来继续做
5. 顺手做笔记与总结

例如你在做聊天服务时：

- 写 Controller
- 写 LLM 调用 Service
- 做流式返回
- 存会话历史
- 碰到 JWT 不会，再补 JWT
- 碰到 Redis 不会，再补 Redis

这样学得最快，也最牢。
