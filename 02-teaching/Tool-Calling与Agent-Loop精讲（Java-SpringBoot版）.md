# Tool Calling 与 Agent Loop 精讲
## Java / Spring Boot 版

> 这份文档要解决的是一个关键升级：
> **从“模型会回答”升级到“模型会借助工具做事”。**

---

# 1. 为什么 Tool Calling 是 Agent 的核心能力

如果没有工具，大模型本质上只能：

- 根据上下文生成文本
- 做一定程度的推理
- 给建议、做总结、写内容

但它做不了很多真实动作，比如：

- 查数据库
- 查天气
- 调企业 API
- 创建任务
- 发起审批
- 搜索知识库
- 调用你自己的业务函数

所以 Agent 能不能“干活”，关键不在于它多会说，而在于：

> **它能不能在合适的时候，选择合适的工具，并把工具结果纳入后续决策。**

这就是 Tool Calling 的价值。

---

# 2. Tool Calling 到底是什么

Tool Calling 不是说“模型真的自己去执行函数”。

更准确地说，它是：

> **模型根据你提供的工具描述，输出一个结构化的调用意图；真正执行工具的是你的程序。**

所以完整过程通常是：

1. 你告诉模型有哪些工具可用
2. 模型判断这次要不要调用工具
3. 如果要，就输出工具名 + 参数
4. 后端解析这次调用
5. 后端真正执行工具
6. 把工具执行结果再给模型
7. 模型基于结果生成最终回答，或继续下一轮工具调用

这就是 Agent Loop 的基础骨架。

---

# 3. 一个最简单的例子

假设你给模型定义了一个工具：

- `getWeather(city)`

用户说：

- “上海今天适合穿薄外套吗？”

模型可能不会直接回答，而是先输出：

```json
{
  "tool": "getWeather",
  "arguments": {
    "city": "上海"
  }
}
```

然后后端执行：

- 调天气 API
- 得到气温、降雨、风力信息

再把工具结果回传给模型，例如：

- 上海，17℃，小雨，风力 3 级

模型再生成最终回答：

- “今天偏凉又有小雨，薄外套可以，但建议带伞。”

你会发现：

- 模型负责决定“查什么”
- 工具负责获取真实世界数据
- 后端负责把这两者串起来

---

# 4. Tool Calling 和普通函数调用有什么不同

普通程序里的函数调用是：

- 你在代码里明确写死调用谁
- 传什么参数由程序员决定

而 Tool Calling 里：

- 工具清单是你提供的
- 但这次调不调、调哪个、参数怎么填，是模型根据上下文判断的

所以它的难点不只是“调用”，而是：

- 如何描述工具
- 如何约束参数
- 如何处理错误
- 如何控制权限
- 如何防止模型乱调

---

# 5. Tool Schema 为什么关键

你可以把 tool schema 理解成“给模型看的工具说明书”。

它通常会告诉模型：

- 工具名字
- 工具功能
- 参数列表
- 参数类型
- 哪些参数必填
- 每个参数的含义

如果 schema 写得模糊，模型就很容易：

- 选错工具
- 传错参数
- 参数字段缺失
- 值格式不对

## 5.1 好 schema 的特点

一个好的工具 schema 通常：

- 名字明确
- 描述具体
- 参数语义清楚
- 限制尽量明确
- 不给模型太多歧义空间

例如：

- `searchDocuments(query, knowledgeBaseId)`

通常就比：

- `search(data)`

强得多。

---

# 6. 不是工具越多越好

很多人一做 Agent，就想把所有工具都丢给模型。

这通常会带来几个问题：

- 模型更难选工具
- 错误调用概率上升
- prompt 更长，成本更高
- 工具边界重叠，决策混乱

所以更合理的原则是：

> **只给当前场景真正需要的工具。**

比如：

- 知识库问答场景，就给检索类工具
- 任务管理场景，就给任务 / 日历 / 提醒工具
- 企业助手场景，再逐步扩展 CRM、审批、文档类工具

工具系统应该“按场景裁剪”，不是“越全越高级”。

---

# 7. Agent Loop 是什么

Agent Loop 就是：

> **模型判断 → 工具执行 → 结果回填 → 模型继续判断**

这样反复循环，直到任务完成或达到停止条件。

## 7.1 一个典型循环

1. 用户提出任务
2. 模型判断要不要调用工具
3. 如果需要，输出 tool call
4. 后端执行工具
5. 把工具结果以 tool message 形式回传
6. 模型继续思考
7. 可能继续调下一个工具
8. 最终生成回答

## 7.2 为什么叫 loop

因为很多任务不是一次工具调用就结束。

例如：

- 先查知识库
- 再查数据库
- 再总结
- 再生成任务

这就天然是一个循环。

---

# 8. 为什么 Agent Loop 不能无限跑

这是非常现实的问题。

如果你不加控制，模型可能会：

- 连续反复调用工具
- 在错误状态下循环
- 一直试图“继续做点什么”
- 无意义地消耗 token 和外部 API 成本

所以你必须在系统层面设置边界。

## 8.1 常见停止条件

- 最大工具调用次数
- 最大循环轮数
- 总超时时间
- 达到最终答案标志
- 关键错误出现后中断
- 需要人工确认时暂停

工程里一定要记住：

> **Agent 的自由度必须被框住。**

---

# 9. Java / Spring Boot 怎么实现 Tool Calling

从工程结构上，你通常可以这样拆。

## 9.1 ToolDefinition

描述工具信息：

- name
- description
- parameter schema
- access policy

## 9.2 ToolExecutor

真正执行工具。

例如：

- WeatherToolExecutor
- SearchDocsToolExecutor
- CreateTaskToolExecutor

每个执行器负责：

- 参数校验
- 调内部服务或第三方 API
- 统一返回结果

## 9.3 ToolRegistry

用来注册当前系统有哪些工具可用，并按名字找到执行器。

## 9.4 AgentService

负责整个 loop：

- 组装上下文
- 把工具 schema 发给模型
- 解析 tool call
- 调用对应执行器
- 回填工具结果
- 判断是否继续

## 9.5 ToolResultFormatter

负责把工具执行结果格式化成统一消息结构，便于再交给模型。

---

# 10. 参数校验为什么特别重要

模型给的参数不一定可靠。

常见问题包括：

- 缺字段
- 字段名不对
- 类型不匹配
- 日期格式错
- 枚举值非法
- 填了业务上无权限的 ID

所以你不能因为“这是模型输出”就直接信任。

后端必须做：

- schema 校验
- 业务校验
- 权限校验
- 默认值处理
- 错误提示或修复策略

这和前端传参数一样，永远不能裸信。

---

# 11. 工具结果怎么回传给模型

工具执行完后，不是简单返回给用户就结束了。

很多时候你需要把结果回传给模型，让它继续基于结果推理。

例如：

- 搜到了 3 条资料
- 查到了客户信息
- 创建任务成功，任务 ID 为 xxx

这些结果都应该变成模型上下文的一部分。

## 11.1 回传时要注意什么

- 信息尽量结构化
- 不要塞太多无关数据
- 不要把超长原始响应全丢回去
- 关键字段要明确
- 必要时做摘要

因为工具返回值也会占 token。

# 12. 什么时候需要人工介入

不是所有工具都适合让 Agent 直接执行。

有些操作风险很高，比如：

- 发邮件
- 删除数据
- 提交审批
- 转账
- 修改生产环境配置

这种时候常见做法是：

> **模型先生成动作建议，最终执行前由人工确认。**

这就是 Human in the Loop。

## 12.1 一个实用原则

你可以把工具分成几类：

### 只读工具
例如：
- 查天气
- 查知识库
- 查客户资料

风险低，可自动执行。

### 低风险写工具
例如：
- 创建草稿
- 新建待办
- 保存中间记录

可在限制条件下自动执行。

### 高风险写工具
例如：
- 发正式邮件
- 删除资源
- 扣费
- 提交外部系统变更

默认需要人工确认。

---

# 13. 工具错误是常态，不是异常事件

真实环境里，工具失败太常见了：

- 外部 API 超时
- 参数不合法
- 权限不足
- 第三方返回 500
- 数据不存在
- 依赖系统限流

所以 Agent 设计不能假设“一次成功”。

## 13.1 常见处理策略

### 直接失败
适合不可恢复错误。

### 给模型返回错误信息，让它调整
例如：
- 参数缺失，请补充城市名
- 当前用户无权访问该资源

### 自动重试
适合瞬时错误，如超时、429。

### 人工介入
适合高风险或多次失败的场景。

# 14. Tool Calling 和 Workflow 的关系

这两者不是对立的。

更实用的理解是：

- Workflow 负责大流程
- Tool Calling 负责让某些步骤更灵活

例如一个“客户跟进助手”流程：

1. 先查客户基本信息
2. 再查历史沟通记录
3. 再生成跟进建议
4. 最后生成邮件草稿

这里大框架可以是 workflow，
而其中“到底要不要再查一次知识库、要不要查 CRM、要不要起草任务”，可以交给模型用工具来判断。

所以很多好系统其实是：

> **Workflow 做主干，Agent 做局部智能决策。**

---

# 15. 怎么防止 Agent 乱用工具

这是工程里非常重要的一环。

## 15.1 权限控制

不是所有用户都能调所有工具。

例如：

- 普通员工不能查管理员知识库
- 普通客服不能调用财务系统接口
- 不同租户之间不能互相访问资源

## 15.2 工具白名单

按场景只暴露必要工具。

## 15.3 参数约束

比如：

- knowledgeBaseId 必须来自当前会话可访问列表
- ownerId 只能是当前用户自己
- 日期范围不能超过某个限制

## 15.4 结果审计

记录：

- 谁触发了什么工具
- 调了几次
- 参数是什么
- 成功还是失败
- 最终影响了什么资源

---

# 16. 日志为什么在 Agent 系统里更重要

因为你不只是在调试代码，还在调试：

- 模型决策
- 工具选择
- 参数生成
- 循环次数
- 错误恢复

至少建议记录：

- requestId
- conversationId
- 用户问题
- 提供给模型的工具列表
- 模型选了哪个工具
- 工具参数
- 工具返回摘要
- 循环次数
- 最终回答
- 异常信息

没有这些日志，你很难知道问题出在：

- 模型理解错了
- schema 写坏了
- 参数校验过严
- 工具本身挂了
- loop 没收住

# 17. 一个你该建立的心智模型

你可以把 Tool Calling / Agent Loop 想成这样：

- **LLM** 负责“判断下一步最可能该做什么”
- **工具** 负责“真正接触外部世界或业务系统”
- **后端** 负责“限制、执行、校验、记录、收敛”

所以真正可靠的 Agent，不是“模型自己很神”，而是：

> **模型的决策能力 + 后端的控制能力 + 工具的执行能力**

三者一起工作。
