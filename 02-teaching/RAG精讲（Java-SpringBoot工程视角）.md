# RAG 精讲
## Java / Spring Boot 工程视角

> 这份文档的目标，是把 RAG 这件事讲清楚：
> **它为什么存在、核心链路是什么、Java / Spring Boot 项目里应该怎么落地、哪些地方最容易做坏。**

---

# 1. 为什么会需要 RAG

很多人刚接触大模型时，会有一个自然想法：

- 模型不是很聪明吗？
- 那我把问题直接问它不就行了？

但真实工程里，这很快会遇到几个问题：

- 模型不知道你公司的内部知识
- 模型的训练数据不是实时的
- 模型可能会一本正经地编
- 你需要答案有出处，而不是“像对的”

所以才需要 RAG。

RAG = Retrieval-Augmented Generation，检索增强生成。

它的核心思想非常朴素：

> **先查资料，再让模型基于资料回答。**

这样做的意义是：

- 把回答建立在外部知识上，而不是纯靠模型参数记忆
- 降低幻觉
- 让答案可追溯
- 支持私有知识库
- 让模型获取更新的数据

---

# 2. RAG 适合解决什么问题

RAG 特别适合这类场景：

- 企业知识库问答
- 文档检索问答
- FAQ 系统
- SOP / 操作手册问答
- 合同、制度、规范解读
- 基于上传文件的问答
- 代码库、接口文档辅助问答

你可以把它理解成：

> **凡是“答案应该来自某些资料”的问题，优先考虑 RAG。**

而不太适合直接靠 RAG 解决的，通常是：

- 强事务型流程
- 纯逻辑计算类任务
- 复杂执行型 Agent 编排
- 不依赖资料的创作任务

---

# 3. RAG 的完整链路是什么

一个典型的 RAG 系统通常包含两条主链路。

## 3.1 离线链路：建库

也就是把资料处理进系统。

步骤通常是：

1. 文档导入
2. 文本提取
3. 文档清洗
4. 切块
5. embedding 向量化
6. 存入向量库
7. 保存元数据

## 3.2 在线链路：问答

也就是用户发问时，系统怎么回答。

步骤通常是：

1. 用户提问
2. query 向量化
3. 向量检索
4. metadata 过滤
5. rerank 重排（可选）
6. 拼接上下文
7. 交给 LLM 生成答案
8. 返回结果和引用来源

你要先把这两条链路彻底记住。

RAG 的本质不是“向量库”，而是：

> **一套从文档处理到检索再到回答生成的完整系统。**

---

# 4. 文档导入：第一步就可能做坏

很多人以为“把 pdf 丢进去就行”，但文档导入其实决定了后面的上限。

## 4.1 文本提取

不同类型文档，处理方式不同：

- PDF
- Word
- Markdown
- HTML
- Excel
- 扫描件图片

要注意几个现实问题：

- PDF 可能提取顺序错乱
- 表格内容可能丢失结构
- 标题层级可能丢失
- 扫描件可能需要 OCR
- 一些文档里有页眉页脚噪音

如果这一步提取错了，后面检索效果通常也会差。

## 4.2 文档清洗

在正式切块前，往往要先处理：

- 空白字符
- 页码噪音
- 重复页眉页脚
- 多余换行
- 乱码

工程上不要小看这个步骤。

很多“RAG 不准”的问题，其实不是模型问题，而是原始文本就脏。

# 5. 切块为什么关键

切块（chunking）是 RAG 成败的关键点之一。

因为模型不能每次都吃完整文档，太长、太贵、也不利于精准检索。

所以你需要把文档拆成较小的片段。

## 5.1 为什么不能切得太大

切得太大，会导致：

- 一个 chunk 包含多个主题
- 检索命中后噪音太多
- prompt 被无关信息占满
- 成本上升

## 5.2 为什么也不能切得太小

切得太小，会导致：

- 语义上下文断裂
- 模型看不到完整含义
- 检索到的片段不够回答问题

## 5.3 常见切块策略

### 固定长度切块

按字数 / token 数切。

优点：
- 实现简单

缺点：
- 容易把语义切断

### 段落切块

按自然段落切。

优点：
- 更符合语义边界

缺点：
- 段落长度差异可能很大

### 标题层级切块

按文档结构切，比如：
- 一级标题
- 二级标题
- 小节

优点：
- 很适合规范文档、知识库、说明书

### 语义切块

按语义完整性切，不完全依赖字符长度。

优点：
- 效果通常更好

缺点：
- 实现更复杂

## 5.4 overlap 是什么

overlap 就是 chunk 之间的重叠部分。

它的作用是：

- 避免关键信息刚好被切断
- 保留相邻片段的上下文连续性

但 overlap 也不能过大，不然会：

- 重复信息变多
- 存储成本上升
- 检索结果高度重复

---

# 6. Embedding 到底是什么

embedding 是 RAG 的核心技术基础之一。

你可以把它理解成：

> **把一段文本映射成一个高维向量，让“语义相近”的文本在向量空间里彼此更接近。**

比如：

- “怎么重置密码”
- “忘记密码后如何找回”

表面字面不同，但语义很接近。

embedding 的目标就是让系统能识别这种“语义近似”。

## 6.1 它不是摘要

embedding 不是把文本变成一句话摘要。

它更像是：

- 一组数字表示
- 供计算机做相似度比较

## 6.2 query 和 document 都要 embedding

RAG 里通常会：

- 先把文档 chunk 做 embedding，存起来
- 用户提问时，再把 query 做 embedding
- 用 query 向量去找最相似的文档向量

## 6.3 相似度计算

常见会用：

- cosine similarity
- 内积
- 欧式距离

你不一定要一开始深挖数学，但要知道这些分数本质上是在衡量“语义接近程度”。

---

# 7. 向量库在做什么

很多人以为向量库很神秘，其实它干的事很明确：

- 存储向量
- 建索引
- 快速相似度检索
- 结合 metadata 做过滤

## 7.1 你要存的不只是向量

实际工程里，一条向量记录通常还会带：

- chunk 文本内容
- 文档 ID
- 文档标题
- 段落位置
- 标签
- 权限范围
- 上传用户
- 创建时间

这些 metadata 非常重要。

因为真实场景往往不是“全库随便搜”，而是：

- 只搜某个知识库
- 只搜当前用户有权限的文档
- 只搜某个部门资料
- 只搜某个时间范围内的内容

## 7.2 没有 metadata filter 的 RAG 很容易出事

比如：

- 搜出来别人的私有文档
- 搜出来无关项目资料
- 搜出来已过期版本

所以检索不只是“语义相似”，还必须“业务可控”。

---

# 8. 检索阶段到底在做什么

用户发出问题后，系统在线链路通常是这样的：

1. 理解 query
2. 做 query embedding
3. 到向量库召回 topK 候选
4. 用 metadata 做过滤
5. 必要时做 rerank
6. 选出最适合的若干片段
7. 拼进 prompt
8. 交给 LLM 回答

## 8.1 topK 是什么

topK 就是先召回前 K 个最相似的 chunk。

K 太小：
- 容易漏掉关键资料

K 太大：
- 噪音增多
- rerank 成本变高
- prompt 太长

## 8.2 metadata filter

这一步是业务控制的核心。

例如：

- knowledgeBaseId = 当前知识库
- tenantId = 当前租户
- visibility = public
- ownerId = 当前用户

## 8.3 rerank 重排

向量检索召回的是“语义近似候选”，但不一定最适合最终回答。

所以常见做法是：

- 先粗召回一批
- 再用 rerank 模型或规则重排
- 选出更相关的少量片段

rerank 的意义是：

- 提高精度
- 减少噪音
- 让送给 LLM 的上下文更干净

---

# 9. 拼接上下文不是越多越好

很多 RAG 初学者有个误区：

- 检索到的片段越多越保险

其实常常相反。

上下文塞太多，会带来几个问题：

- token 成本上升
- 无关信息干扰模型
- 模型抓不到重点
- 多段内容互相矛盾

所以一个好的 RAG 系统，要做的是：

> **把最相关、最够用、最可信的资料送给模型。**

而不是把一堆相似片段全糊上去。

## 9.1 拼接时要注意什么

建议关注：

- 是否保留文档标题
- 是否标注文档来源
- 是否保留段落编号
- 是否按相关性排序
- 是否去重
- 是否控制总 token

---

# 10. 生成阶段的 prompt 应该怎么设计

RAG 的 prompt 设计和普通聊天不一样。

它通常必须明确要求模型：

- 优先基于提供资料回答
- 如果资料不足，不要编
- 需要引用来源
- 回答不要超出上下文支持范围

## 10.1 一个典型思路

system prompt 里可以强调：

- 你是知识库问答助手
- 仅根据提供资料回答
- 若资料不足，请明确说明
- 在回答中标明引用来源

然后 user prompt 附上：

- 用户问题
- 检索片段列表
- 每段片段的来源信息

## 10.2 为什么这能降低幻觉

因为你把模型的任务从：

- “自由回答”

改成了：

- “基于给定证据回答”

这会显著提升可靠性。

# 11. RAG 里最常见的失败点

这部分非常重要，因为很多人做了 RAG 但效果烂，往往不是单点问题，而是链路上某个环节坏了。

## 11.1 文档质量差

- 原始文档脏
- OCR 错误多
- 提取顺序错乱

## 11.2 切块策略差

- chunk 太大
- chunk 太小
- 语义被切断
- 重叠不合理

## 11.3 embedding / 检索策略不合适

- query 和文档表达不一致
- 召回 topK 太小或太大
- 缺少 metadata filter

## 11.4 prompt 设计差

- 没约束模型必须基于资料
- 没要求“资料不足就说不知道”
- 没要求引用来源

## 11.5 评测缺失

- 觉得“看起来还行”就上线
- 没有样例集
- 不知道哪类问题答得差

# 12. Java / Spring Boot 里怎么拆 RAG 模块

从工程视角，一个 RAG 系统可以这样拆。

## 12.1 DocumentController

负责：

- 上传文档
- 查询文档列表
- 查看处理状态
- 触发重建索引

## 12.2 DocumentParseService

负责：

- 文本提取
- 文档清洗
- 格式标准化

## 12.3 ChunkService

负责：

- 按策略切块
- 生成 chunk 元数据

## 12.4 EmbeddingService

负责：

- 调 embedding 模型
- 返回向量结果

## 12.5 VectorStoreService

负责：

- 保存向量
- 删除向量
- 执行相似度检索
- metadata filter

## 12.6 RetrievalService

负责：

- 处理 query
- 召回候选 chunk
- 重排 / 去重 / 裁剪
- 返回最终上下文

## 12.7 RagChatService

负责：

- 组装 prompt
- 注入检索片段
- 调 LLM
- 生成最终回答
- 附带引用来源

你会发现，RAG 并不是一个“函数”，而是一个子系统。

# 13. RAG 和数据库是什么关系

这也是常见误区。

很多人以为有了向量库，就不需要 MySQL 了。

不是。

## 13.1 MySQL / PostgreSQL 负责什么

结构化数据：

- 用户
- 文档
- 知识库
- 权限
- 上传记录
- 文档状态
- 处理任务
- 引用关系

## 13.2 向量库存什么

语义检索所需数据：

- chunk 文本
- embedding 向量
- metadata

所以更准确的理解是：

- **关系型数据库负责业务状态**
- **向量库负责语义召回**

两者是配合关系，不是替代关系。

# 14. 怎么评估一个 RAG 系统做得好不好

不能只靠“我试了几个问题感觉还行”。

至少要建立基本评测方法。

## 14.1 可以从三个层面看

### 召回层

问：
- 正确资料有没有被召回？
- topK 里是否包含关键 chunk？

### 生成层

问：
- 最终回答是否忠于资料？
- 是否有编造？
- 是否引用了正确来源？

### 系统层

问：
- 响应速度是否可接受？
- token 成本是否过高？
- 不同文档类型效果是否稳定？

## 14.2 最少要有一套样例集

你应该准备：

- 一组常见问题
- 每个问题期望引用哪些资料
- 期望答案大概长什么样

这样你后续改切块、换 embedding、调 topK，才知道是变好了还是变坏了。

# 15. RAG 的工程升级方向

当你做出一个基础版 RAG 后，后面通常会继续优化这些方向：

## 15.1 Query Rewrite

把用户问题先改写成更适合检索的表达。

例如用户问得太口语、太短、代词太多时，先规范化 query。

## 15.2 Multi-query Retrieval

为一个问题生成多个检索表达，扩大召回面。

## 15.3 Hybrid Search

结合：

- 向量检索
- 关键词检索

有些场景下，精确术语和数字特别重要，单纯向量检索不一定够。

## 15.4 Rerank

先粗召回，再精排。

## 15.5 Citation / Grounding

让回答和引用绑定得更明确。

## 15.6 Access Control

把权限严格绑定到检索链路，避免资料泄漏。

# 16. 一个你该建立的心智模型

你可以把 RAG 想成这样：

- **文档处理层**：把原始资料变成可检索知识块
- **语义检索层**：从大量知识块里找出相关证据
- **回答生成层**：让模型基于证据作答
- **业务控制层**：管理权限、范围、来源和日志

所以 RAG 的关键不是“用了没”，而是：

> **你的证据链干不干净、检索准不准、回答是不是忠于证据。**
