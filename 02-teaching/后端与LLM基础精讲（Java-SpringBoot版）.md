# 后端与 LLM 基础精讲
## Java / Spring Boot 版

> 这份文档的目标，不是让你背术语，而是让你真正建立一个清晰脑图：
> **Java 后端在 AI 项目里到底负责什么，LLM 到底是什么，Spring Boot 又该怎么把它们串起来。**

---

# 1. 先建立一个总图：AI 项目不是“模型 = 全部”

很多人一接触 AI 开发，会下意识觉得：

- 重点就是模型
- 调通 API 就差不多了
- Prompt 写好就能做产品

这其实不对。

真正的 AI 项目里，模型只是其中一层。

一个完整系统一般长这样：

1. **前端 / 客户端**
   - 聊天界面
   - 文件上传
   - 任务状态页
   - 管理后台

2. **Java / Spring Boot 后端**
   - 接收请求
   - 鉴权
   - 管理用户和会话
   - 管理文件、知识库、任务
   - 调用模型
   - 调用工具
   - 存储结果

3. **LLM 层**
   - 理解输入
   - 生成内容
   - 决定是否调用工具
   - 根据上下文推理

4. **数据层**
   - MySQL / PostgreSQL：结构化数据
   - Redis：缓存、会话、状态
   - 向量库：知识检索
   - 对象存储：文件

5. **外部工具层**
   - 搜索服务
   - 企业内部 API
   - 日历、邮件、文档系统
   - 自定义函数

6. **工程保障层**
   - 日志
   - 监控
   - 限流
   - 重试
   - 成本控制
   - 权限控制

所以你要先记住一句话：

> **AI 项目不是“把模型接进来”，而是“让模型嵌入现有系统并可控地工作”。**

---

# 2. Java / Spring Boot 在 AI 项目里的角色

很多教程会让你误以为：

- 后端只是一个 API 转发器
- Spring Boot 只负责把前端请求转发给模型

实际上完全不是。

Java / Spring Boot 在 AI 项目里，通常承担的是“系统骨架”的角色。

## 2.1 它负责业务规则

例如：

- 哪些用户能访问哪些知识库
- 哪些人能调用哪些工具
- 哪些任务需要人工审批
- 文件上传后走什么处理链路
- 某类请求是否允许使用高成本模型

这些都不应该交给模型决定，而应该由后端明确控制。

## 2.2 它负责状态管理

模型本身不擅长管理长期状态。

所以后端要管理：

- 用户信息
- 对话会话
- 消息历史
- 工具调用记录
- 任务执行状态
- 费用统计
- 文档处理状态

## 2.3 它负责系统集成

AI 产品很少是一个“孤立聊天框”。

后端通常要接：

- 用户系统
- 权限系统
- 数据库
- 文件系统
- 第三方服务
- 企业内部业务 API

Spring Boot 在这种整合型场景里非常合适。

## 2.4 它负责安全和稳定性

比如：

- 模型请求超时怎么办
- 第三方 API 429 怎么办
- 用户恶意刷接口怎么办
- 工具调用失败怎么回滚
- 某个任务执行一半中断怎么办

这些都是后端工程问题，不是 prompt 能解决的。

---

# 3. 为什么 Spring Boot 适合做 AI 后端

## 3.1 成熟

Spring Boot 的最大优势不是“最新”，而是“稳”。

它适合做：

- 业务 API
- 权限系统
- 配置管理
- 数据库访问
- 文件处理
- 定时任务
- 企业系统集成

而这些正是 AI 产品真正需要的大部分底座。

## 3.2 分层清晰

AI 项目一旦变复杂，如果没有分层，很快就会变得不可维护。

Spring Boot 很适合把系统拆成：

- Controller：接收请求
- Service：业务逻辑
- Repository / Mapper：数据访问
- Client：外部模型和第三方服务调用
- Tool：工具能力封装
- Domain / Entity / DTO：数据模型

这会让 AI 逻辑不至于和业务逻辑搅成一锅粥。

## 3.3 适合企业化落地

很多真实 AI 项目最后都会碰到这些问题：

- 权限复杂
- 审计要求高
- 接口规范严格
- 系统需要上线维护
- 需要日志和监控

在这类事情上，Spring Boot 比“临时拼一个脚本服务”强很多。

# 4. LLM 到底是什么

这是最基础但最容易被说得很玄的一部分。

LLM = Large Language Model，大语言模型。

从工程角度，你可以先把它理解成：

> **一个根据上下文生成下一段文本的概率模型。**

它并不是“真的理解人类世界”的存在，也不是数据库，更不是搜索引擎。

## 4.1 它是怎么工作的

超简化理解：

1. 你给它一段上下文
2. 它把这段上下文拆成 token
3. 它基于训练得到的模式，预测下一个 token 最可能是什么
4. 一个 token 接一个 token 地生成结果

你可以把它想象成：

- 不是“知道答案”
- 而是“根据上下文，继续写出最像合理答案的内容”

这也是为什么它会：

- 有时非常聪明
- 有时一本正经地胡说八道

## 4.2 它为什么看起来像“会思考”

因为语言本身包含了大量模式。

当模型学到了足够多的语言模式、推理轨迹和任务格式后，它在很多问题上会表现得像在思考。

但工程上你不要神化它。

更好的理解是：

> **它擅长模式拟合、上下文推理、文本生成，不擅长天然保证事实正确。**

# 5. Token 是什么

这个概念必须搞清楚，因为它会直接影响：

- 成本
- 响应速度
- 上下文长度
- 会话设计

## 5.1 Token 不是字，也不是单词

模型不直接按“汉字个数”或“英文单词个数”工作。

它处理的是 token。

一个 token 可能是：

- 一个字
- 半个词
- 一个单词
- 一个标点组合
- 某种常见字符串片段

所以你写得越长、上下文越多，token 通常越多。

## 5.2 Token 为什么重要

因为大模型调用通常按 token 计费，而且上下文窗口也有限。

这意味着：

- 输入越长，越贵
- 输出越长，越贵
- 历史消息堆太多，会拖慢和拉高成本
- 检索片段塞太多，可能反而让效果下降

## 5.3 工程上的意义

你以后做系统时必须经常思考：

- 这段系统提示词是否太长
- 历史会话是否要裁剪
- 检索内容是否太冗余
- 是否能先摘要再传给模型

# 6. Context 是什么

Context 就是“模型当前看到的全部输入上下文”。

它通常包括：

- system prompt
- user prompt
- 历史对话
- 检索出来的文档片段
- 工具返回结果
- 结构化状态信息

模型不会神奇地记得所有过去的事。

它只能基于这一次请求里你给它的 context 来工作。

所以很多人误以为：

- “模型怎么突然忘了我前面说的话？”

本质原因通常是：

- 你没把相关信息带进去
- 或者上下文太长，被裁掉了
- 或者重要信息埋得太深，模型没抓住

## 6.1 上下文窗口

每个模型都有 context window，也就是能处理的上下文上限。

超过上限通常会：

- 报错
- 被截断
- 成本变高
- 表现变差

所以你不能无脑把所有历史对话都塞进去。

## 6.2 为什么上下文管理是工程核心

AI 产品里很大一部分工作，本质上是在做：

> **有限上下文预算下的信息编排。**

你要决定：

- 哪些历史消息保留
- 哪些做摘要
- 哪些信息必须高优先级出现
- 检索内容放多少
- 工具结果展示到什么粒度

这就是上下文工程。

# 7. Prompt 到底是什么

Prompt 不是“咒语”，它本质上是你给模型的任务说明书。

你可以把 prompt 理解成：

- 背景说明
- 任务目标
- 输入数据
- 输出格式要求
- 约束条件

一个清晰的 prompt，通常比一个“灵感式 prompt”更可靠。

## 7.1 Prompt 的常见组成

### system prompt

定义角色和全局规则，比如：

- 你是一个企业知识库问答助手
- 回答必须基于提供的资料
- 如果资料不足，就明确说不知道
- 输出用 JSON 格式

### user prompt

用户本轮真实输入。

### developer / hidden instructions（视平台而定）

某些系统里还有开发者层规则，用来约束模型行为。

## 7.2 好 prompt 的特点

一个好的 prompt 通常：

- 目标明确
- 输入边界清楚
- 输出格式清楚
- 约束条件具体
- 少废话
- 可复用

## 7.3 坏 prompt 的典型问题

- 目标不清
- 一句话里混了多个任务
- 没有输出格式
- 让模型“自由发挥”太多
- 没限定信息来源

---

# 8. System Prompt 为什么这么重要

在多轮对话和 Agent 系统里，system prompt 很像“系统人格 + 工作规则 + 安全边界”的合集。

它会影响：

- 模型语气
- 任务边界
- 输出规范
- 工具使用偏好
- 拒答策略
- 资料引用要求

## 8.1 一个实用理解

如果 user prompt 是“这一单怎么做”，
那么 system prompt 更像是：

> “你是谁、你该怎么做事、哪些规则不能破。”

## 8.2 工程上的注意点

system prompt 不是越长越好。

太长会导致：

- 成本上升
- 重点模糊
- 重复约束
- 模型抓不到最关键规则

所以更好的原则是：

- 核心规则前置
- 结构清晰
- 只写必要约束
- 避免相互冲突

# 9. 结构化输出为什么关键

很多初学者喜欢让模型直接返回自然语言。

但工程里，很多时候你真正需要的是：

- JSON
- 固定字段
- 枚举值
- 可解析结果

因为系统后面可能还要：

- 存库
- 渲染 UI
- 驱动流程
- 调下一个工具
- 触发审批

如果输出不稳定，后面全会乱。

## 9.1 一个典型例子

比如你要做“任务提取器”，自然语言输出可能是：

- “好的，我建议明天下午联系客户并准备一份报价。”

但系统真正想要的是：

```json
{
  "action": "create_task",
  "title": "联系客户并准备报价",
  "due_date": "2026-03-17 15:00",
  "priority": "high"
}
```

两者都“像答案”，但只有第二种能驱动系统。

## 9.2 所以要养成习惯

以后只要涉及：

- 流程驱动
- UI 渲染
- 结果存档
- 工具执行

优先思考：

> **能不能让模型输出结构化结果？**

# 10. 幻觉是什么，为什么会出现

幻觉就是：

> **模型生成了听起来像真的，但实际上不可靠、错误、编造或未经验证的内容。**

## 10.1 为什么会幻觉

根本原因是：

- 模型目标是“生成最像合理回答的文本”
- 不是“保证事实正确”

常见诱因包括：

- 问题超出模型可靠知识边界
- 提供的信息不足
- prompt 没限制信息来源
- 要求它一定给答案
- 检索结果本身不准

## 10.2 工程上怎么降低幻觉

不是靠祈祷，而是靠系统设计：

- 资料型问题优先走 RAG
- 明确要求“资料不足就说不知道”
- 要求给出引用来源
- 对关键任务增加规则校验
- 对高风险场景加入人工审核

# 11. Java 后端接 LLM 的基本调用链路

从系统视角，一个最基础的 AI 接口大概长这样：

1. 前端发起请求
2. Controller 接收请求
3. Service 组装 prompt / context
4. LLM Client 调用模型 API
5. 拿到结果后解析
6. 记录日志 / 会话 / token 使用量
7. 返回给前端

## 11.1 一个典型模块拆法

### Controller
负责：

- 接收 HTTP 请求
- 参数校验
- 调用 Service
- 返回响应

### ChatService
负责：

- 读取历史消息
- 构造 system prompt
- 拼接 user 输入
- 调用 LLM Client
- 保存本轮结果

### LlmClient
负责：

- 调第三方模型 API
- 处理认证信息
- 处理超时 / 重试
- 解析响应结构

### MessageRepository / Mapper
负责：

- 保存会话消息
- 查询历史记录

### TokenUsageService
负责：

- 统计输入输出 token
- 计算成本
- 做额度控制

你会发现，模型调用本身只是其中一环。

# 12. 为什么“会话管理”是聊天系统核心

很多人做聊天 demo，只是把用户这一句话丢给模型。

但一旦做多轮系统，就必须有会话管理。

你至少要管理：

- conversationId
- userId
- message 列表
- 消息角色（system/user/assistant/tool）
- 创建时间
- token 消耗
- 摘要信息

## 12.1 为什么不能把所有历史全带上

因为：

- 太贵
- 太慢
- 太长会超过上下文上限
- 噪音太多会降低效果

## 12.2 常见策略

### 策略 1：只保留最近 N 轮
优点简单。缺点容易丢长期信息。

### 策略 2：旧消息做摘要
把老内容压缩成摘要，只保留关键事实。

### 策略 3：重要信息单独存 memory
把“长期有效信息”单独存数据库，再按需取回。

这三种通常会组合使用。

# 13. 流式输出是什么，为什么重要

你会发现现在大多数 AI 产品都喜欢“边生成边显示”。

这就是流式输出。

## 13.1 为什么它体验更好

因为用户会感觉：

- 系统正在工作
- 延迟更低
- 响应更自然

即使总耗时差不多，流式也通常比“等 10 秒一下全出来”更舒服。

## 13.2 后端怎么理解这件事

你要知道：

- 模型可能逐段返回 token
- 后端需要把这些片段持续转发给前端
- 前端要实时拼接显示

常见实现方式：

- SSE
- WebSocket
- 分块 HTTP 响应

如果你是 Java / Spring Boot 路线，SSE 是一个很常见的起点。

# 14. 错误处理为什么在 AI 项目里尤其重要

AI 系统的失败点比普通 CRUD 多得多。

你可能会遇到：

- 模型接口超时
- 第三方限流
- 输出 JSON 格式错误
- 工具调用参数缺失
- 检索为空
- 文件解析失败
- 中间状态没存好

## 14.1 所以你必须把失败当常态

后端设计时不要想“应该一次成功”，而要想：

- 如果失败了怎么提示用户
- 能不能自动重试
- 重试几次
- 要不要降级
- 中间状态怎么恢复
- 哪些异常要报警

这就是工程思维。

# 15. 为什么 AI 项目更需要日志

因为你调试的不是只有代码，还有：

- prompt
- 历史上下文
- 检索结果
- 模型输出
- 工具调用链

如果没有日志，你会非常痛苦。

## 15.1 至少要记录什么

建议至少记录：

- 请求 ID
- 用户 ID / 会话 ID
- 使用的模型
- 输入 token / 输出 token
- 请求耗时
- 是否命中工具调用
- 检索到哪些文档
- 最终是否成功
- 错误原因

## 15.2 但别乱记

注意隐私和敏感信息，别把所有用户原文无脑打日志。

对于敏感场景：

- 做脱敏
- 做摘要记录
- 控制日志访问权限

# 16. 一个你应该建立的正确心智模型

把一个 AI 后端系统想成这样：

- **Spring Boot** 是骨架
- **数据库 / Redis** 是记忆与状态
- **LLM** 是推理与生成引擎
- **工具系统** 是手和脚
- **RAG** 是外接知识
- **日志 / 监控** 是体检系统
- **前端** 是用户交互界面

而你作为工程师要做的事，不是“把最强模型接进来”，而是：

> **把这些组件以稳定、可控、可维护的方式组合起来。**

这才是真正的 AI Agent 工程。
