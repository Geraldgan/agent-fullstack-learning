# iOS 工程师 → AI Agent 全栈工程师
## Java / Spring Boot 主线版学习手册

> 适合：已有 iOS 开发经验，希望系统补齐 Java 后端、AI Agent、全栈工程能力的人。
>
> 目标不是只会调大模型 API，而是能独立设计、开发、部署一个可用的 AI Agent 系统。

---

# 1. 先说结论：你到底要转成什么样的人

你不是单纯从 **iOS 工程师** 转成 **Java 后端工程师**，也不是只会调 LLM API 的“AI 应用开发者”。

你的目标应该是：

> **能独立设计、开发、部署、优化一个 AI Agent 系统的人。**

这个系统通常包括：

- **客户端入口**：App、Web、IM、桌面端
- **后端服务**：Java / Spring Boot，负责业务逻辑、权限、数据管理、工作流编排
- **AI 能力层**：LLM、RAG、Tool Calling、Memory、Planning、Agent Runtime
- **数据层**：MySQL / PostgreSQL、Redis、向量库、对象存储
- **工程层**：日志、监控、鉴权、限流、部署、成本控制、评测

所以你的升级路径不是：

**iOS → 学点 AI**

而是：

**iOS 工程师 → 会后端 → 会 AI Agent → 会交付完整系统**

## 阶段 B：学会把 LLM 接入 Java 系统

### 目标
从传统后端升级成 AI 后端。

### 你要学什么

#### LLM 的基本工作方式
你不用一开始研究 Transformer 数学细节，但要懂：
- 模型接收的是 token，不是“自然理解”
- 模型是根据上下文预测下一个 token
- 它不是数据库，不是搜索引擎
- 它有概率性，有幻觉
- 它对上下文非常敏感

#### API 调用
你必须会：
- 调 chat completion / responses 类接口
- 传 system / user / assistant 消息
- 控制 temperature、max tokens
- 处理流式输出
- 处理超时、重试、限流

#### Prompt 设计
要掌握：
- system prompt 的角色
- task prompt 的结构化表达
- 给模型明确输入 / 输出格式
- few-shot 示例
- 约束条件
- 如何减少废话和跑偏

#### 结构化输出
你要会让模型输出：
- JSON
- 固定字段
- 指定 schema
- 可解析格式

因为工程里不能只看自然语言，必须可消费。

#### 上下文管理
你要懂：
- 为什么不能把所有历史都喂进去
- 怎么裁剪历史消息
- 怎么做摘要
- 怎么保留重要状态
- 如何做会话级 memory

## 阶段 C：掌握 RAG 和知识库

### 目标
让系统不只靠模型记忆，而是能查资料再回答。

### 什么是 RAG
RAG = Retrieval-Augmented Generation。

也就是：

> 先检索相关资料，再把资料给模型，让模型基于资料回答。

这比“直接问模型”更可靠，尤其适合：
- 企业知识库
- 文档问答
- FAQ
- SOP 流程
- 资料密集型场景

### RAG 的核心流程
1. 文档导入
2. 文档切块
3. 向量化 embedding
4. 存入向量库
5. 用户提问
6. 检索相关片段
7. 拼接上下文
8. 交给模型生成答案

### 你要掌握的知识点

#### 文档切块
为什么要切？
因为整个文档太长，模型上下文贵，而且检索不准。

要理解：
- chunk size
- overlap
- 按段落切
- 按标题切
- 按语义切

#### Embedding
embedding 不是“摘要”，而是把文本映射到向量空间，便于做相似度搜索。

你要懂：
- 向量表示语义相似性
- query 和 document 都要 embedding
- 常见相似度：cosine similarity

#### 向量库
你要知道它在干嘛：
- 存向量
- 快速召回相似内容
- 支持 metadata 过滤

#### 检索优化
RAG 不是“接个向量库”就结束。还要学：
- topK
- metadata filter
- rerank 重排
- hybrid search 混合检索
- query rewrite 查询改写

#### 引用与可追溯
好的 RAG 系统应该：
- 告诉用户信息来自哪里
- 引用原文片段
- 尽量减少模型乱编

# 5. Workflow 和 Agent 的区别

这是非常容易混淆的点。

## 5.1 Workflow 是什么
Workflow 是你预先设计好的流程。

例如：
1. 用户上传简历
2. 系统提取文本
3. 调模型做摘要
4. 再调模型做岗位匹配
5. 输出结构化结果

这叫工作流。步骤固定，控制权主要在程序。

## 5.2 Agent 是什么
Agent 是在目标驱动下，模型能够：
- 理解任务
- 选择工具
- 决定下一步
- 根据结果调整策略
- 最终完成任务

例如：
“帮我整理这个客户的背景，并生成一封跟进邮件。”

Agent 可能会：
1. 查 CRM
2. 查历史邮件
3. 搜公司背景
4. 总结要点
5. 起草邮件
6. 等待人工确认

步骤不是全固定的。

## 5.3 什么时候用 Workflow，什么时候用 Agent

### 用 Workflow 的场景
- 流程明确
- 步骤固定
- 可控性要求高
- 稳定性优先
- 成本敏感

### 用 Agent 的场景
- 任务开放
- 需要动态选择工具
- 步骤不固定
- 更看重灵活性

### 真正的工程实践
大多数可用系统不是纯 Agent，而是：

> **Workflow + 局部 Agent 能力**

也就是：
- 大流程由程序控制
- 某些复杂决策点交给模型

这是最实用的。

# 6. Tool Calling / Function Calling 是什么

这是 Agent 工程的核心能力之一。

它的意思不是模型“真的自己会用工具”，而是：

> 模型根据你的工具描述，决定是否要调用某个函数，并输出调用参数；真正执行工具的是你的程序。

比如你定义工具：
- searchDocs(query)
- getWeather(city)
- createTask(title, dueDate)

模型看到用户问题后，可能会说：
“我需要调用 getWeather(city='上海')”

然后你的程序：
1. 解析这个工具调用
2. 真正执行 Java 方法 / HTTP 请求
3. 把结果返回给模型
4. 模型再生成最终回答

### 为什么它重要
因为没有 tool calling，模型只能“说”；有了 tool calling，它才能“做”。

### Java / Spring Boot 中怎么落地
你最终做的通常是：
- 定义工具协议
- 写工具执行器
- 让模型返回结构化 tool call
- 后端执行
- 把结果塞回上下文

这就是 Agent 的骨架之一。

# 7. Memory 到底是什么

很多人一说 memory，就想得很神秘。其实工程上它通常分几类：

## 7.1 短期记忆
当前会话里的上下文：
- 最近几轮对话
- 当前任务状态
- 当前已知用户目标

## 7.2 长期记忆
跨会话保存的信息：
- 用户偏好
- 常用项目
- 历史任务
- 长期背景资料

## 7.3 工作记忆
任务执行中的中间状态：
- 已调用过哪些工具
- 已拿到哪些信息
- 哪些子任务已完成

### 关键点
Memory 不是“把所有聊天记录全存起来”。真正重要的是：
- 存什么
- 怎么检索
- 什么时候更新
- 什么时候清理

否则只会越来越脏。

# 8. Planning 是什么

Planning 就是任务拆解。

比如用户说：
“帮我做一个竞品调研并输出总结。”

系统可以拆成：
1. 确定竞品列表
2. 获取产品信息
3. 分维度比较
4. 生成总结
5. 输出文档

### 规划能力的价值
复杂任务不是靠一次 prompt 解决，而是：
- 拆任务
- 逐步执行
- 检查结果
- 修正路径

### 但要注意
不是所有系统都要做“超智能规划”。很多场景下：
- 规则化任务拆解
- 模板化步骤
- 少量 LLM 决策

已经够用了。

# 11. Java / Spring Boot 在 AI Agent 里的位置

很多教程默认 Python，但你走 Java 完全没问题。

## 11.1 Java 的优势
- 企业级生态成熟
- Spring Boot 工程化强
- 权限、数据库、事务、接口治理成熟
- 更适合接企业内部系统
- 团队协作和生产部署更稳

## 11.2 Java 的挑战
- AI 新生态很多先出在 Python
- 部分示例代码偏少
- 某些新框架支持可能不如 Python 快

## 11.3 正确姿势
不要执着于“所有 AI 代码都必须 Java 写”。

更现实的架构是：

### 方案 A：纯 Java 主服务
- Spring Boot 做主后端
- 直接调模型 API
- 集成 RAG、工具调用、业务系统

适合大多数应用型项目。

### 方案 B：Java 主服务 + Python AI 子服务
- Java 负责主业务
- Python 负责某些 AI 能力模块
- 通过 HTTP / MQ 调用

适合后期复杂化场景。

你当前阶段，建议先走 **方案 A**。

# 13. 你最容易踩的坑

## 坑 1：以为会调用模型 API 就算会 AI
不是。那只是最基础的一步。

## 坑 2：沉迷 Prompt，不补后端能力
结果做出来全是 demo，不能上线。

## 坑 3：过早追逐复杂 Agent
一上来就做 autonomous multi-agent，最后一地鸡毛。

## 坑 4：忽视数据和检索质量
RAG 做不好，回答就会烂。

## 坑 5：只关注生成效果，不关注系统稳定性
真实项目里，稳定、可追踪、可重试比“偶尔很惊艳”更重要。

## 坑 6：学太散
今天看 LangChain，明天看 AutoGen，后天再看某个新框架，最后脑子里没有主线。

# 21. 知识精讲 Part 1：后端与 LLM 基础

这一部分开始，把前面路线图里出现的核心概念讲透一点。

## 21.1 为什么 AI Agent 工程师必须懂后端
因为 Agent 不是一个“会聊天的页面”就结束了。

真正上线后的系统，至少要处理这些问题：
- 用户身份是谁
- 谁能访问哪些知识库
- 文件存在哪里
- 对话历史怎么保存
- 工具调用有没有权限
- 模型调用失败怎么重试
- 成本怎么统计
- 日志怎么追踪

这些问题本质上都是后端问题。

所以你如果只会：
- 写 prompt
- 调模型 API
- 做个前端聊天窗口

那你能做 demo，但很难做产品。

## 21.2 Spring Boot 在 AI 系统里通常负责什么
一个典型 Spring Boot AI 项目，通常会负责：

### 1）用户与权限
- 登录
- Token 校验
- 用户角色控制
- API 权限隔离

### 2）业务数据管理
- 用户表
- 会话表
- 消息表
- 文档表
- 知识库表
- 工具调用日志表

### 3）AI 调度层
- 组织 prompt
- 调用 LLM API
- 处理 tool call
- 选择模型
- 管理上下文

### 4）集成外部系统
- 企业内部接口
- 搜索系统
- CRM
- 邮件系统
- 文件存储
- 向量库

### 5）稳定性与运维能力
- 超时
- 重试
- 熔断
- 限流
- 监控
- 日志审计

所以你可以把 Spring Boot 理解成：

> **AI Agent 系统的业务中枢。**

## 21.3 一个最小可用 AI 后端长什么样
如果你要做最小版本，结构通常会像这样：

```text
Client (Web/App)
   ↓
Controller
   ↓
ChatService
   ↓
LLMClient
   ↓
Model API
```

再稍微真实一点，就会变成：

```text
Client
   ↓
Controller
   ↓
ChatService
   ├── SessionService
   ├── PromptBuilder
   ├── ToolDispatcher
   ├── KnowledgeService
   └── LLMClient
          ↓
       Model API
```

这时候你已经不是“调一下接口”，而是在做系统了。

## 21.4 什么是 LLM API 接入
本质上就是：

1. 你构造消息上下文
2. 发请求给模型
3. 拿回模型输出
4. 必要时继续循环

通常请求内容会包括：
- system prompt
- user message
- 历史消息
- 工具定义
- 输出格式要求

### 一个很重要的认知
模型不是“知道你业务”的。

它只知道：
- 你传给它的上下文
- 它训练中学过的泛化知识
- 当前这一轮的工具描述

所以系统能力强不强，很多时候不取决于模型多聪明，而取决于：
- 你给了什么上下文
- 你怎么设计工具
- 你怎么组织流程

## 21.5 Token 到底是什么
你可以先把 token 理解成：

> 模型处理文本时使用的最小计算单位。

不是一个汉字一定等于一个 token，也不是一个英文单词一定等于一个 token。
它取决于分词方式。

### 为什么 token 重要
因为它直接影响：
- 成本
- 响应速度
- 上下文长度
- 截断风险

### 工程上的实际影响
如果你：
- 历史消息太长
- 文档检索片段太多
- prompt 写得很啰嗦

那就会：
- 更贵
- 更慢
- 更容易跑偏

所以 AI 工程不是只追求“信息越多越好”，而是追求：

> **给模型足够的信息，但不要给无效信息。**

## 21.6 Prompt 到底是什么
Prompt 不只是“提问”。
在工程里，prompt 更像是：

> 你给模型下达的一份任务说明书。

一个好的 prompt，通常包含：
- 角色说明
- 任务目标
- 输入数据
- 约束条件
- 输出格式
- 示例

### 一个简单结构
```text
你是一个企业知识库助手。
任务：根据给定资料回答问题。
要求：
1. 只能基于资料回答
2. 不确定就明确说不知道
3. 输出用 JSON，字段为 answer、confidence、citations
资料：...
问题：...
```

### 为什么很多人 prompt 写不好
因为他们写的是“想法”，不是“指令”。

坏 prompt 常见问题：
- 目标不明确
- 输入不完整
- 输出格式不清楚
- 约束太弱
- 一句话塞太多要求

## 21.7 结构化输出为什么特别重要
如果模型返回一大段自然语言，给人看可以，但给程序处理就很痛苦。

所以真实工程里常常要求模型输出：
- JSON
- 指定字段
- 枚举值
- 数组结构

比如：

```json
{
  "intent": "create_task",
  "title": "整理周报",
  "priority": "high"
}
```

这样后端就能直接解析，再执行对应逻辑。

### 结构化输出的本质价值
是把：
- “模型会说话”

变成：
- “模型能参与程序流程”

这一步非常关键。

## 21.8 流式输出为什么重要
你在客户端应该很容易理解这个问题。

如果用户发起一次请求，要等 8 秒才整段返回，体验会很差。
但如果模型一边生成一边返回，用户会感觉系统“在工作”。

常见方式：
- SSE
- WebSocket

### 对后端来说意味着什么
你要处理：
- 流式转发模型输出
- 连接中断
- 前端取消请求
- token 增量拼接
- 最终结果落库

所以它不是单纯“开个流”就结束，而是整条链路都要考虑。

## 21.9 多轮对话的本质
很多人以为多轮对话就是“把历史消息拼起来”。
这是最基础的一层，但不够。

真正的问题是：
- 历史多少轮要保留
- 哪些信息重要
- 哪些废话可以删掉
- 什么时候做摘要
- 用户偏好放哪里

### 一个现实做法
你可以把上下文分成几层：
- system instruction
- recent messages
- task state
- retrieved knowledge
- tool results

这样组织比简单拼接聊天记录更稳。

# 22. 知识精讲 Part 2：RAG、Tool Calling、Agent Loop

## 22.1 为什么只靠模型记忆不够
因为模型有三个天然限制：
- 不知道你企业私有数据
- 训练数据可能过时
- 容易幻觉

所以如果问题依赖：
- 内部文档
- 新数据
- 实时数据
- 可验证知识

就不能只靠模型脑补，必须让它“查”。

这就是 RAG 的价值。

## 22.2 RAG 的本质不是“外挂知识”，而是“先找再答”
本质流程是：

```text
用户问题
  ↓
检索相关资料
  ↓
把资料拼到上下文
  ↓
模型基于资料生成答案
```

关键点在于：

> 模型不是直接回答，而是“看完资料再回答”。

## 22.3 RAG 最容易出问题的地方
### 1）切块不合理
切太大：召回噪声高。
切太小：上下文碎片化。

### 2）检索不准
用户问法和文档写法不一致，导致召回偏差。

### 3）资料塞太多
相关性低的片段一多，模型就容易乱。

### 4）没有引用
用户不知道答案从哪来，也不敢信。

所以一个好 RAG 系统的优化方向，通常不是先换更大模型，而是先优化：
- chunking
- retrieval
- rerank
- prompt grounding

## 22.4 Tool Calling 的本质
Tool Calling 的本质不是“模型会执行代码”，而是：

> 模型负责决定“该调用什么工具、传什么参数”；程序负责真正执行。

这是非常重要的边界。

### 为什么这样设计
因为：
- 模型擅长理解自然语言和做决策
- 程序擅长执行确定性动作

两者分工才稳。

## 22.5 一个工具调用的最小闭环
```text
用户：帮我查上海天气
   ↓
模型判断：需要 getWeather(city="上海")
   ↓
后端执行工具
   ↓
工具返回天气数据
   ↓
模型整合成最终回复
```

### 这里至少涉及 4 个模块
1. 工具定义
2. 工具调度器
3. 工具执行器
4. 二次生成回复

所以 Tool Calling 其实已经是一个小型系统设计问题。

## 22.6 Agent Loop 是什么
Agent Loop 可以先粗暴理解成：

> 模型思考一次 → 决定要不要调工具 → 执行工具 → 把结果喂回去 → 再决定下一步。

它是一个循环，而不是一次性调用。

例如：
1. 用户让系统分析一家公司的背景
2. 模型先搜索公司官网
3. 再搜索新闻
4. 再总结风险点
5. 最终输出分析

这就不是一次请求能完成的，而是多步循环。

## 22.7 为什么很多 Agent 项目做着做着就失控
因为 loop 一旦开放，就容易出这些问题：
- 无限循环
- 工具乱调
- 成本失控
- 结果不可解释
- 错误难追踪

所以工程上必须加约束：
- 最大步数
- 工具白名单
- 参数校验
- 超时控制
- 人工确认点
- 日志追踪

### 记住一句话
**越智能的系统，越需要护栏。**

## 22.8 Human in the loop 为什么重要
很多业务动作不能让模型自动拍板，比如：
- 发邮件
- 改数据库关键数据
- 删除文件
- 对外发送消息
- 创建审批流程

这类动作最好的做法是：
- 模型先产出建议
- 人确认
- 再执行

这样更符合真实业务系统要求。
