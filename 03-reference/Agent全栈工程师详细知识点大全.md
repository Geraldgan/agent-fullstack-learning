# 一、Agent 全栈工程师到底是什么

Agent 全栈工程师，不等于“会调大模型 API 的前端工程师”，也不等于“会写 Java 接口的后端工程师”。

它更接近一种复合型角色：

- 能做 Web 产品界面
- 能写业务后端
- 能接入大模型
- 能设计工具调用、知识库、记忆、工作流
- 能把系统部署上线并长期维护

如果用一句话概括：

**Agent 全栈工程师 = 全栈开发能力 + AI 应用能力 + 工程落地能力。**

你当前的 iOS 背景，其实有不少可迁移能力：
- UI 和交互意识
- 网络请求经验
- 工程结构意识
- 调试能力
- 代码质量意识

你要补的主要是：
- Web 前端生态
- Java 后端实战
- Agent 架构与实现
- 云端部署与运维

## 2.1 Web 基础

### 1. HTTP / HTTPS
#### 是什么
浏览器和服务器之间通信的协议。

#### 为什么要学
前端和后端的连接本质上就是 HTTP 请求。你不懂协议，就很难定位接口问题、登录问题、跨域问题。

#### 重点掌握
- 请求方法：GET / POST / PUT / DELETE
- 状态码：200、400、401、403、404、500
- 请求头 / 响应头
- Content-Type
- Authorization
- Cookie
- HTTPS 比 HTTP 多了 TLS 加密

#### 在 Agent 项目里怎么用
- 前端调用 `/api/chat/send`
- 登录时带 token
- 文件上传时用 multipart/form-data
- SSE / 流式输出也建立在 HTTP 上

#### 学到什么程度
- 能抓包看请求
- 能看懂接口失败原因
- 能区分前端问题和后端问题

---

### 2. DNS、域名、端口
#### 是什么
- DNS：把域名解析成 IP
- 端口：一台机器上不同服务的入口

#### 为什么要学
部署时你一定会接触：
- 前端 3000
- 后端 8080
- MySQL 3306
- Redis 6379
- 域名访问和 Nginx 代理

#### 学到什么程度
- 知道为什么浏览器访问 `localhost:3000`
- 知道域名为什么能访问服务器
- 知道反向代理的大致作用

---

### 3. Cookie / Session / LocalStorage / SessionStorage
#### 是什么
浏览器侧和服务端侧保存状态的不同方式。

#### 重点区别
- Cookie：浏览器自动携带，小型数据
- Session：服务端保存会话状态
- LocalStorage：浏览器本地长期存储
- SessionStorage：页签级短期存储

#### 在 Agent 项目里怎么用
最常见是：
- JWT 放 LocalStorage 或 Cookie
- 用户主题配置放 LocalStorage

#### 学到什么程度
- 知道登录态放哪儿
- 知道它们的安全差异

---

### 4. 跨域 CORS
#### 是什么
浏览器同源策略限制不同域名 / 端口间请求。

#### 为什么要学
前端跑 `localhost:3000`，后端跑 `localhost:8080`，这就是跨域。

#### 重点掌握
- 什么叫同源
- 为什么后端要配置 CORS
- 预检请求 OPTIONS

#### 学到什么程度
- 知道跨域报错怎么解决

# 四、Agent / AI 详细知识点

---

## 4.1 大模型基础

### 1. token
理解：
- 模型不是按“字”理解，而是按 token 处理
- token 数影响成本和上下文长度

### 2. context window
即模型一次能看的上下文范围。

### 3. temperature
控制输出的随机性。

### 4. system / user / assistant
对话消息通常由这几种角色组成。

### 5. hallucination
模型胡说八道。

#### 为什么要理解
这决定你不能盲目信任模型，必须加入工具、检索、约束和校验。

---

## 4.2 Prompt Engineering

### 1. 角色设定
告诉模型它是谁。

### 2. 任务明确化
告诉模型要做什么，不要让它猜。

### 3. 输出格式约束
如：
- 输出 JSON
- 输出 markdown
- 输出固定字段

### 4. few-shot
给几个例子帮助模型按预期输出。

### 5. 防跑偏
加入边界条件：
- 不知道就说不知道
- 不要编造来源
- 必须引用检索内容

#### 学到什么程度
- 能写出稳定可复用的 prompt 模板

---

## 4.3 Tool Calling

### 是什么
模型不直接回答，而是先调用某个工具拿到事实，再组织答案。

### 你要理解的完整流程
1. 给模型一组工具定义
2. 模型判断是否需要工具
3. 模型返回工具名和参数
4. 后端执行工具
5. 工具结果回给模型
6. 模型生成最终答案

### 重点难点
- 工具 schema 设计
- 参数校验
- 工具执行失败处理
- 防止模型乱传参数

### 第一批建议工具
- 天气
- 搜索
- 日历 / 待办
- 数据库查询

---

## 4.4 RAG

### 是什么
先检索资料，再让模型基于资料回答。

### 为什么重要
它能显著降低胡说，提高基于私有知识回答的能力。

### 核心流程
1. 文档导入
2. 文本清洗
3. 切片 chunking
4. embedding
5. 向量检索
6. 拼接上下文
7. 模型回答
8. 返回引用来源

### 重点知识点
#### 文档切片
为什么切：
- 文档太大不能整篇塞给模型
- 切片后更容易检索命中局部内容

#### embedding
把文本变成向量，用来做相似度检索。

#### rerank
进阶概念，先理解就行。

#### 上下文拼接
检索到多个片段后，如何组织进 prompt。

### 学到什么程度
- 能做一个基础知识库问答系统
- 知道回答为什么要附来源

---

## 4.5 Memory

### Agent 的记忆分层
#### 1. 短期记忆
当前对话上下文。

#### 2. 长期记忆
用户偏好、长期信息、历史关系。

#### 3. 外部记忆
数据库、向量库、文件系统。

### 为什么重要
Agent 如果完全没有记忆，就只能“一次性回答”。

### 学到什么程度
- 能设计“会话历史”和“用户偏好”两层记忆

---

## 4.6 Workflow / Agent 架构

### 1. 单 Agent
一个模型直接处理请求。

### 2. Router
根据用户意图分流到不同处理逻辑。

### 3. Planner-Executor
先规划，再执行。

### 4. 多步任务流
例如：
- 搜索资料
- 提炼重点
- 生成报告

### 5. Human-in-the-loop
关键步骤要求人工确认。

### 学到什么程度
- 能把任务拆成几个阶段
- 能设计状态流转

---

## 4.7 Agent 工程化

### 1. 日志与追踪
要记录：
- 用户输入
- prompt
- 模型输出
- 工具调用过程
- 错误信息

### 2. 成本控制
理解：
- token 很贵
- 历史上下文不能无限堆
- 模型可按任务分层使用

### 3. 回退机制
例如：
- 主模型失败时切备用模型
- 工具失败时给出保守回答

### 4. 安全边界
- 权限控制
- 用户隔离
- 工具访问限制
- 敏感操作二次确认
