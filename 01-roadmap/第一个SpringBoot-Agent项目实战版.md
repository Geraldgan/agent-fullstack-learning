# 第一个 Spring Boot Agent 项目实战版

> 这份文档的目标，不是讲概念，而是给你一条可落地的项目主线：
> **做一个真正能跑起来的 AI Agent 产品原型。**

我建议你的第一个项目，不要一上来就做“万能超级 Agent”。

更稳的方向是做一个：

> **企业知识助手 / 工作台 Agent**

它既能覆盖 AI Agent 的核心能力，又不会复杂到无法收拾。

---

# 1. 这个项目应该长什么样

建议做成一个 Web 项目，包含：

## 前端

- 聊天界面
- 文档上传页
- 知识库管理页
- 任务状态页
- 登录页

## 后端（Spring Boot）

- 用户登录鉴权
- 会话管理
- 消息管理
- 文档上传与处理
- RAG 检索
- Tool Calling
- Agent Loop
- 日志与统计

## 数据层

- MySQL：用户、会话、消息、文档、任务
- Redis：会话缓存、限流、临时状态
- 向量库：知识库检索
- 对象存储：原始文件

# 3. 第一阶段：最小可运行版本（MVP）

一开始不要做太大。

你先做出最小版本，只包含这些能力：

## 3.1 功能范围

- 用户登录
- 创建聊天会话
- 发送消息给模型
- 显示模型回复
- 保存聊天记录
- 上传文档
- 基于文档做问答

也就是说，第一版其实是：

> **聊天 + 文档问答**

先把它跑通。

## 3.2 这一版暂时不做的东西

先不急着做：

- 多工具复杂 Agent
- 多角色协作
- 自动规划大任务
- 很复杂的前端美化
- 精细化监控系统

因为你第一阶段最重要的是：

> **建立一条完整闭环。**

---

# 4. 第二阶段：加入 Tool Calling

当 MVP 跑通后，再加入工具能力。

## 4.1 先接哪些工具

建议先接 2-3 个最简单又最能体现 Agent 感的工具：

- `searchKnowledgeBase(query)`
- `getTaskList(userId)`
- `createTodo(title, dueDate)`

这样你就能做出一个“不是只会聊天，而是会查、会创建任务”的系统。

## 4.2 为什么这样选

因为这些工具：

- 业务边界清楚
- 数据容易模拟
- 演示效果直观
- 风险较低

先不要一上来就接高风险工具，比如：

- 发邮件
- 删除数据
- 真正外部审批提交

---

# 5. 第三阶段：加入任务状态和执行记录

到了这一步，项目会明显更像一个系统，而不只是 demo。

## 5.1 建议加入

- 工具调用日志
- 任务执行状态
- 每轮 Agent loop 记录
- 错误信息记录
- token 使用量统计

## 5.2 为什么重要

因为 AI 产品的“可交付感”来自这些地方：

- 可追踪
- 可审计
- 可恢复
- 可解释

这也是企业系统和玩具 demo 的区别。

# 7. 后端模块怎么拆

一个适合新手又够清晰的拆法如下。

## 7.1 auth

负责：

- 登录
- JWT
- 用户身份校验

## 7.2 chat

负责：

- 会话创建
- 消息发送
- 聊天记录存储
- 流式输出

## 7.3 document

负责：

- 文件上传
- 文档解析
- 文档状态管理

## 7.4 rag

负责：

- chunking
- embedding
- vector search
- 检索上下文构造

## 7.5 tool

负责：

- 工具定义
- 工具执行
- Tool Registry

## 7.6 agent

负责：

- Agent loop
- 工具调用编排
- 最终答案生成

## 7.7 common / infra

负责：

- 全局异常处理
- 日志
- 配置
- HTTP 客户端
- 工具类

# 10. 一个现实可行的开发顺序

下面这个顺序很重要，能帮你少踩坑。

## 第 1 步：先搭基础后端骨架

完成：

- Spring Boot 项目初始化
- MySQL 连接
- Redis 连接
- 用户登录/JWT
- 全局异常处理
- 基础日志

## 第 2 步：做最基础聊天接口

完成：

- 创建会话
- 发送消息
- 调 LLM
- 存消息记录

## 第 3 步：做前端聊天页

完成：

- 会话列表
- 聊天输入框
- 消息显示
- 接口联调

## 第 4 步：做文档上传和 RAG

完成：

- 文件上传
- 文档解析
- chunk + embedding
- 检索问答

## 第 5 步：做工具调用

完成：

- 定义 2-3 个工具
- 做 Tool Registry
- 跑通 Agent loop

## 第 6 步：补日志、状态、错误处理

完成：

- tool call log
- token 统计
- 失败重试
- 状态展示

这个顺序的优点是：

- 每一步都能看到成果
- 每一步都能联调验证
- 不会一上来就堆太多复杂度

# 13. 常见误区

## 误区 1：项目功能越多越厉害

错。第一项目最重要的是：

- 主线清晰
- 架构稳
- 能讲清楚
- 真能跑

## 误区 2：一开始就追最复杂的 multi-agent

这很容易把项目做散。

## 误区 3：前端完全不管

最后系统只能 Postman 演示，产品感太弱。

## 误区 4：没有日志和状态记录

出了问题根本查不到。

## 误区 5：只做对话，不做业务落点

那就还是聊天 demo，不够像真正的 Agent 产品。
